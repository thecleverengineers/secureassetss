# AGENTS.md

## Cursor Cloud specific instructions

SecureAsset Enterprise is a single full-stack app (not a monorepo): a React 18 + Vite SPA (`src/`), an Express 5 + Socket.IO API (`server/src/`), MongoDB via Mongoose, and optional PM2 cron workers (`scripts/`). Standard commands live in `package.json` `scripts` and setup docs in `README.md` / `LOCAL_SETUP.md`; prefer those instead of re-deriving commands.

### Runtime / toolchain (already provisioned by the update script + VM snapshot)
- Node.js `24.18.0` and npm `11.16.0` are required (`engines` in `package.json` + `.npmrc` has `engine-strict=true`; a `preinstall` runtime check hard-fails on wrong versions). The VM has a system `/exec-daemon/node` (v22) earlier in `PATH`; the correct Node is selected via nvm and prepended to `PATH` in `~/.bashrc`. If a command fails a version check, run `nvm use 24.18.0` first (or start a login shell).
- MongoDB 8.0.26 (the version this release is validated against) is installed. It is NOT managed by systemd here — start it manually per session (see below). Data persists in the VM snapshot at `/var/lib/mongodb`.

### Start services (per session; do NOT put these in the update script)
1. MongoDB: `mongod --dbpath /var/lib/mongodb --bind_ip 127.0.0.1 --port 27017 --logpath /var/log/mongodb/mongod.log --logappend` (run in a tmux session; it does not daemonize). Verify with `npm run db:check`.
2. App (dev): `npm run dev` starts the API on `:5000` (nodemon) and Vite on `:5173` concurrently. Open http://localhost:5173. Vite proxies `/api`, `/socket.io`, and `/uploads` to the API.

### One-time database setup (data persists in the snapshot)
`.env` is generated by `npm run setup:local` (random JWT secrets; `STORAGE_DRIVER=local`, `DEMO_OTP=123456`, `PAYMENT_AUTO_APPROVE=false`). After a fresh DB, seed + migrate:
`npm run seed`, then `npm run migrate:document-vault`, `npm run migrate:advanced-rental`, `npm run db:indexes`.

- GOTCHA: `npm run dev` uses nodemon directly and does NOT run the automatic startup migrations that `npm start` does. The CMS-driven data (property types, landlord plans, area units) comes from `migrate:advanced-rental`. If the "Property Type" dropdown in the property wizard is empty/stuck loading, that migration has not been run — run it, then hard-reload the SPA (site config is cached client-side and only refetched on reload).
- Seed demo accounts (password `Demo@123`): `admin@secureasset.in`, `manager@secureasset.in`, `tenant@secureasset.in` (has Landlord Mode), `surveyor@secureasset.in`. Every account is a Tenant; Landlord/Surveyor modes are toggled via the top-right mode dropdown.

### Lint / test / build
- Lint: `npm run lint:server` (ESLint over `server`, `scripts`); frontend type-check is `./node_modules/.bin/tsc --noEmit`.
- Test: `npm run test` (`node --test test/*.test.js`). Some tests (SPA-routing) require a prior `npm run build` because they serve `dist/index.html`.
- Build: `npm run build` outputs to `.frontend-releases/<release>/` and atomically points the `dist` symlink at it.
